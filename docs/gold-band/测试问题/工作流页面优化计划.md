当前现状：
1. 单击节点只切换选中态，主要影响下方产物/附件列表；没有产物或附件时反馈很弱。
2. 节点右键菜单中存在查看详情、查看日志、查看会话、复制节点 id、从该节点重跑等功能，核心信息入口过度依赖右键。
3. 查看详情、查看日志、查看会话目前都偏原始 JSON/JSONL 文本查看，缺少结构化理解；会话内容没有区分 runtime 进度事件和 provider 原始流。
4. Round 顶部同时存在打开日志、导出日志按钮，其中外层导出日志入口没有形成清晰的信息架构。

优化目标：
重构 Round 工作流菜单、节点详情、会话与日志信息页。当前项目处于开发阶段，本次直接做破坏式更新，不保留旧 JSON 查看器、旧下方信息流主入口或外层日志导出按钮的灰度兼容逻辑。

1. 单击工作图节点后，从右侧滑出节点抽屉；抽屉左侧外部提供两个垂直 tab：查看详情、查看会话。抽屉打开时默认进入查看详情。
2. 查看详情 tab 不再展示 node.json 原始 JSON，改为可视化信息界面：节点信息、attempt 信息、状态、时间、产物、附件、会话可用状态等。
3. 详情中的 artifact、attachment 列表提供详情跳转。点击后滑出二级抽屉展示完整内容，二级抽屉左上提供返回按钮，返回上一级节点详情抽屉。
4. 查看会话 tab 用横向 tab 区分 `progress.events` 和 `raw.stream`：
   - `progress.events` 展示 runtime/provider 进度事件。
   - `raw.stream` 展示 provider 原始 stdout/stderr stream envelope。
5. 查看日志从会话中分离。顶部只保留“打开日志”按钮，删除外层“导出日志”按钮；日志抽屉内部提供导出能力。
6. 日志页展示系统关键排障日志，一条日志一行，包含时间、类型、节点、阶段、摘要等字段，并支持分页。
7. 日志采用冷热数据策略：
   - 默认检索当前热日志，热查询限制最近约 1000 条，保证页面打开速度。
   - 全量日志保留 30 天，用于深度排障、导出或扩大检索范围。
   - 首版不引入 SQLite；现有 JSONL 已经是一行一条，先基于 JSONL tail、结构化解析和分页实现。后续若需要跨任务全文检索、复杂筛选或大规模索引，再引入 SQLite。
8. 右键菜单保留为低频快捷入口：查看详情、查看会话、查看日志、复制节点 id、从该节点重跑；核心信息浏览必须通过单击节点即可完成。

验收标准：
1. 单击节点直接打开详情抽屉，默认选中查看详情。
2. 节点详情不是 JSON，而是结构化 UI。
3. 产物/附件可从节点详情进入二级抽屉，并可返回。
4. 会话页可切换 `progress.events` 与 `raw.stream`。
5. 顶部不再展示外层导出日志按钮。
6. 日志抽屉内展示分页的一行一条日志，并提供导出入口。
7. 页面默认只检索热日志约 1000 条，并在 UI/文档中说明全量日志保留 30 天。
