现在我们处理下侧边栏的上下文管理tab（现有的“知识库”，更名叫上下文管理）
上下文管理中顶部也是一个tab选择页，提供多类上下文管理，本次我们仅支持profile管理（做i18n,中文叫角色管理，英文叫profile管理（？或其他的））
profile管理的就是工作流中的profile
profile 支持三种来源：内置、用户级与项目级。用户级存储在 `~/.gold-band/context/profiles/<name>-<id>.md`，项目级存储在 `~/.gold-band/projects/<project-id>/context/profiles/<name>-<id>.md`；内置角色由客户端内建资源提供，不写入 `context/profiles` 目录。
内置角色包含方案、开发、审查、测试、验收、清理 6 个角色，workflow 继续使用 profile `id` 做引用，角色名称允许重复。历史已经写入 `context/profiles` 的同名默认角色继续按普通用户角色处理，不做迁移兼容。
profile 管理中，可以搜索、新增、修改、查看、删除 profile。列表拆成“自定义角色 / 内置角色”两个 tab：自定义角色 tab 承载用户级与项目级角色，并独立维护搜索、scope 过滤和分页；内置角色 tab 只展示客户端内建角色，并独立维护自己的搜索状态，不与自定义角色列表串扰。卡片展示名称、id、摘要、scope、创建时间、更新时间；查看时显示完整 Markdown 内容。新增时要求用户填写名称、摘要、完整内容，id 由系统分配为分布式唯一 id；创建时间和更新时间使用系统本地时区的 `YYYY-MM-DD HH:MM:SS`。内置角色可打开并编辑草稿，但不能直接保存覆盖，只能通过“另存为新角色”创建新的用户级或项目级角色；删除仅对非内置角色开放，且若仍被 workflow 模板、任务 workflow 或可继续运行快照引用则必须阻止删除。删除失败原因应直接显示在删除确认弹窗内，明确列出当前仍被哪些引用占用，而不是退回到页面级通用横幅。桌面端的角色增删改必须直接调用 Tauri 命令；浏览器 mock 仅用于非 Tauri 预览模式，不能参与桌面端的前置存在性判断。
工作流定义中，用id做profile的引用，工作流配置面板的profile项做成下拉栏，支持关键字（名称、id、摘要、描述等）索引，下拉栏中每一项展示名称、id、摘要（截取、悬浮展示全文）、创建时间、更新时间，下拉栏默认加载用户目录和当前工作空间的所有profile，选中后下拉栏的当前栏显示名称，名称后跟着一个info图标，悬浮显示名称、id、摘要、创建时间、更新时间。工作流配置面板profile标题，后面跟一个info，悬浮提示，profile可在上下文管理中进行管理，跟一个超链接，点击跳转到上下文管理-->profile管理标签页
