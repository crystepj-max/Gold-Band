1.首页任务列表，去掉导入requirements按钮，只保留创建任务
2.点击创建任务按钮，弹出创建任务弹窗
3.弹窗中目前只支持导入现有需求（支持txt\md格式）
4.导入需求后，需要创建工作流
5.工作流可以新增节点和边，为每个节点配置不同的agent，agent来源为agent管理页面的agent卡片
6.工作流保存后持久化为json，在任务详情页的工作流中可以修改工作流，round详情中的工作流则是每一次工作流的运行时快照
7.节点前后需要有环境的生成和销毁(比如释放agent等)

关于工作流的修改与新增：最好是能做到画布式拖拽生成工作流，是否有现成组件可以使用？
关于工作流中的节点与边：
现有的节点类型现在可能要去掉execute节点，和execute-plan产出节点

节点的本质类型只有三种。1.普通节点 2.是否需要人工check？（人工check逻辑你先留白，本期不实现） 2.是否需要验证AI输出结果？ 2.1 输出的DSL配置（也就是AI要输出的指定类型，目前支持json） 2.2 DSL成功条件（json中指定字段的值） 2.3 成功流向（去哪个节点） 2.4 失败流向（去哪个节点），边可以设置一些属性，比如成功边，失败边，边可以指向其他节点（则节点生成新的attempt），或者new round（则生成新的round），边指向节点时，如果是回路（即A-C--》D--》E-->...-->->B-->A）,在B节点指向A的边中可以提供一个选择，是开启新的session，还是继续A刚刚的session？（这里涉及到环判断，可能比较复杂，如果实现困难，可以先用简单方式，即验证输出结果类型的节点，失败边提供这两个选项即可）
工作流这里可以提供一套默认工作流给用户选择：
plan（人工check节点，逻辑留白） --> 开发（普通节点） --> review（普通+验证输出结果，DSL中输出review是否通过，成功指向测试，失败指向开发，默认是继续开发原session） --> 测试（普通+验证输出结果，DSL中输出test是否通过，默认是继续开发原session） --> 验收（普通+验证输出结果，DSL中输出verify是否通过）

## MVP 实现状态

已实现：
- 任务列表移除独立“导入 requirements”入口，只保留“创建任务”。
- 创建任务 Sheet 支持导入 `.txt` / `.md` requirement，填写 title / description，并在同一流程内创建 workflow。
- 工作流编辑器基于 `@xyflow/react`，支持新增节点、拖拽节点、连接边、配置边类型、目标与 session 策略。
- 节点 Agent 来源为 Agent 管理页已配置且支持的 agent card。
- 默认工作流为 `plan -> dev -> review -> test -> accept`，默认不再生成 `exec` 节点和 `exec-plan` 产物。
- worker 节点支持 JSON 输出验证：`output.kind=json` + `success_condition.path/equals`，并按结果路由 success / failure / invalid。
- 边可指向普通节点、`$end` 或 `$new-round`；指向普通节点时会生成新 attempt，指向 `$new-round` 时打开新 round。
- 任务级 workflow 保存到 `authoring/workflow.json`，run 启动时冻结为 `workflow.snapshot.json`；Round 详情展示运行态快照，不回写后续作者态修改。
- 节点执行前后增加 environment setup / teardown 事件与基础资源清理。

本期留白：
- 人工 check 只在 UI 中作为占位说明，不实现运行时阻塞逻辑。
- 后端 `exec` / `exec-plan` 兼容保留，避免破坏既有 workflow 与测试；新建工作流 UI 不再默认生成。
- JSON 条件只支持简单 dot path，不实现完整 JSONPath / JSON Schema 编辑器。
